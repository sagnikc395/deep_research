# Deep Research Agent

A multi-stage research agent built on [smolagents](https://github.com/huggingface/smolagents)
and HuggingFace Inference. Takes a research query from the CLI and produces a structured
Markdown report by coordinating a pipeline of specialised agents.

## Architecture

```
User Query (CLI)
      |
      v
+------------------+
|     Planner      |  LLM call -> produces: goal + list of subtopics (JSON)
+------------------+
      |
      v
+------------------+
|  Task Splitter   |  Pure Python -> produces: list[SubTask] dataclass instances
+------------------+
      |
      v
+------------------------------+
|        Orchestrator          |  ThreadPoolExecutor -> spawns N subagents in parallel
|  +----------+ +----------+   |
|  |SubAgent 1| |SubAgent 2| ..|  Each: ToolCallingAgent + DuckDuckGoSearchTool + visit_webpage
|  +----------+ +----------+   |
+------------------------------+
      |
      v
+------------------+
|    Collector     |  LLM call -> synthesises all subagent outputs into one report
+------------------+
      |
      v
  report.md
```

### Stage responsibilities

| Stage | File | Role |
|---|---|---|
| Planner | `planner.py` | Refines the query into a goal and 3-6 focused subtopics |
| Task Splitter | `task_splitter.py` | Converts subtopics into `SubTask` dataclass instances with full agent prompts |
| Orchestrator | `orchestrator.py` | Runs one `ToolCallingAgent` per subtask concurrently via a thread pool |
| Subagents | `tools/web_tools.py` | Each agent uses `DuckDuckGoSearchTool` + `visit_webpage` for web research |
| Collector | `collector.py` | Calls the LLM once more to synthesise all sections into a single Markdown report |
| CLI | `cli.py` | Wires all stages together; handles args, progress output, and file writing |

### Why ToolCallingAgent for subagents, CodeAgent for nothing?

Web browsing is a single-timeline task: search, pick a page, read it, repeat.
It does not require dynamic code synthesis, only structured tool dispatch.
`ToolCallingAgent` with JSON tool calls is the correct fit here per the
[smolagents multi-agent docs](https://huggingface.co/docs/smolagents/en/examples/multiagents).
The manager/orchestrator role is handled in plain Python (ThreadPoolExecutor)
rather than a `CodeAgent` manager, which avoids unnecessary code execution overhead
and gives us deterministic parallel dispatch.

## Setup

```bash
git clone <this-repo>
cd deep_research_agent

python -m venv .venv
source .venv/bin/activate   # Windows: .venv\Scripts\activate

pip install -r requirements.txt

cp .env.example .env
# edit .env and set HF_TOKEN=hf_your_token_here
```

You need a HuggingFace account with Inference API access. The free tier works
for `Qwen/Qwen2.5-72B-Instruct` but may rate-limit parallel requests; reduce
`--workers` to 1 or 2 if you hit 429 errors.

## Usage

```bash
# Basic usage - output goes to research_<timestamp>.md
python cli.py "impact of transformer models on NLP benchmarks"

# Specify output file
python cli.py "quantum computing hardware approaches" --output reports/quantum.md

# Use more subtasks and workers
python cli.py "CRISPR therapeutic applications" --max-subtasks 6 --workers 3

# Different model
python cli.py "federated learning in healthcare" --model Qwen/Qwen2.5-72B-Instruct

# Skip LLM synthesis - dump raw subagent sections directly
python cli.py "your query" --no-synthesise

# Pass HF token inline (overrides .env)
python cli.py "your query" --hf-token hf_xxx
```

### All CLI options

| Flag | Default | Description |
|---|---|---|
| `query` | (required) | Research topic or question |
| `--output / -o` | `research_<ts>.md` | Output file path |
| `--model / -m` | `Qwen/Qwen2.5-72B-Instruct` | HuggingFace model ID |
| `--max-subtasks` | `5` | Cap on number of subtopics the planner may generate |
| `--workers / -w` | `3` | Parallel subagent thread count |
| `--max-steps` | `10` | Max reasoning steps per subagent |
| `--no-synthesise` | off | Skip final LLM synthesis, output raw sections |
| `--hf-token` | env `HF_TOKEN` | HuggingFace API token |

## Output format

Every report is a Markdown file with a YAML front-matter metadata block:

```
---
generated: 2025-06-23 14:30:00
original_query: "impact of LLMs on software engineering jobs"
subtasks_completed: 5/5
---

# <Title>

## Executive Summary
...

## <Subtopic 1>
...

## Key Sources
- https://...
```

## Project structure

```
├── task_splitter.py    # Stage 2: SubTask dataclass + list construction

└── README.md
```

## Rate limiting

HuggingFace Inference API enforces per-model rate limits. With `--workers 3`
and `Qwen/Qwen2.5-72B-Instruct`, you will typically hit no limits. If you do:

- Reduce `--workers` to `1` or `2`
- Use `--max-subtasks 3` to reduce total agent count
- Switch to a smaller model like `Qwen/Qwen2.5-7B-Instruct` for subagents

## Extending

To add a new tool available to all subagents, define it in `tools/web_tools.py`
with the `@tool` decorator and add it to the `tools=[...]` list in
`orchestrator.py:_run_single_subagent`.

To add a new stage (e.g., a deduplication pass between collector and output),
create a new module and wire it into the `main()` function in `cli.py`.
